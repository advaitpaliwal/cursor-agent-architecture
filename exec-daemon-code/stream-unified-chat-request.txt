## Key Architecture Insights

### 1. Model Routing is Server-Side
  The DEFAULT_CONFIG has "gpt-5" as a client-side fallback only.
  The actual model is determined by:
  - model_details (field 5): ModelDetails with modelId, displayName, credentials
  - Agent-level: AgentRunRequest.requested_model (RequestedModel with model_id, max_mode, parameters)
  - Server resolves the model ID to a specific version/endpoint
  - The deprecated should_speculatively_route_gpt5 confirms GPT-5 routing experiments existed

### 2. Model Fallback System
  allow_model_fallbacks (field 37) enables the server to use a different model
  if the requested model is unavailable. number_of_times_shown_fallback_model_warning
  tracks how many times the user has been warned about fallbacks.

### 3. RequestedModel Has Version Variants
  RequestedModel.is_variant_string_representation (field 8) indicates the model_id
  comes from a ModelVariantConfig. This means "gpt-5" might map to different variants
  (like gpt-5.3) based on server-side configuration. The client sends "gpt-5" but
  the server can resolve it to any variant in the model family.

### 4. Context Bank = Encrypted Conversation Cache
  Three fields (session_id, version, encryption_key) implement encrypted
  conversation state caching in Redis. The encryption key is USER-PROVIDED,
  meaning the server cannot read cached conversations without the client's key.
  This is a privacy-by-design feature for long conversations.

### 5. Speculative Summarization
  speculative_summarization_encryption_key (field 79) enables encrypted
  summarization caching. Combined with force_summarization (field 75),
  this allows proactive conversation compression with E2E encryption.

### 6. Best-of-N Trials
  best_of_n_group_id + try_use_best_of_n_promotion implement A/B testing
  where multiple model responses are generated and the best is selected.
  FreeBestOfNPromotion tracks trials_used and trials_remaining.

### 7. YOLO Mode
  enable_yolo_mode + yolo_prompt enable auto-approval of ALL commands.
  The yolo_prompt field allows users to customize YOLO mode behavior
  (e.g., "auto-approve all git commands").

### 8. Spec Mode
  is_spec_mode (field 77) enables "specification mode" - a dedicated mode
  for writing and executing specifications/requirements.

### 9. Review Mode Connection
  disable_edit_file_timeout (field 70) is set "when review mode is active",
  indicating a PR review mode that disables edit timeout restrictions.
